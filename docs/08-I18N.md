# PokerNotes — Internationalization (i18n)

Complete i18n strategy for PokerNotes, supporting **English (en)** and **Spanish (es)** from day one, with architecture that scales to more locales.

---

## 1. Scope & Principles

### Launch locales
- **English (`en`)** — default, primary market
- **Spanish (`es`)** — secondary market, native language of Fran, large live poker community in Spain + LATAM

### Principles
- **English-first**: source strings in English, translated to Spanish
- **URL-based locale routing**: `/en/...` and `/es/...` for SEO
- **No client-side language detection only**: always use URL for SEO reliability
- **Server-rendered translations**: keep client bundle lean
- **Type-safe keys**: no hardcoded string keys in components

---

## 2. Library Choice: `next-intl`

**Rationale:**
- First-class Next.js 15 App Router support
- Works with Server Components (critical for SEO)
- Type-safe with TypeScript
- Rich message formatting (ICU MessageFormat): plurals, dates, numbers
- Works on Cloudflare Pages (via OpenNext) — no Node-only APIs
- Mature ecosystem, active maintenance

**Alternatives considered & rejected:**
- `next-i18next`: older Pages Router, awkward with App Router
- `react-intl`: no built-in Next.js routing integration
- Custom solution: unnecessary reinvention

---

## 3. URL Structure

```
https://pokernotes.app/            → Redirect to /en or /es based on Accept-Language
https://pokernotes.app/en          → English landing
https://pokernotes.app/es          → Spanish landing
https://pokernotes.app/en/tools/pot-odds
https://pokernotes.app/es/tools/pot-odds      → "Calculadora de pot odds"
https://pokernotes.app/en/notes
https://pokernotes.app/es/notes
```

### SEO: `hreflang` tags
Every page renders `<link rel="alternate" hreflang="en" href="..." />` and same for `es`, plus `x-default` pointing to English.

### Language detection
1. Explicit URL locale → use that (overrides everything)
2. User preference cookie → use that if set
3. `Accept-Language` header → match to supported locale
4. Fallback to `en`

### Translation principles for Spanish poker context

- **Casino Spanish**: use "póker" and "poker" interchangeably (both accepted); prefer "poker" for web-searchability

## 6. AI Content in Multiple Languages

**Critical consideration:** AI-structured notes from Workers AI must respect user's language.

### Strategy
1. Detect user's current locale from request
2. Pass locale into AI system prompt
3. AI returns response in that locale

### Updated AI system prompt

```typescript
function buildSystemPrompt(locale: 'en' | 'es'): string {
  const instructions = {
    en: `Respond in English. Write the summary and tendencies in English.`,
    es: `Responde en español. Escribe el resumen y las tendencias en español.`,
  };

  return `
You are a poker player profiling assistant.
When given a raw note about a poker player's behavior:
1. Extract relevant behavioral tags from the approved list
2. Write a concise 1-2 sentence summary
3. Identify preflop and postflop tendencies

${instructions[locale]}

IMPORTANT: Tags are always returned in English (technical IDs), 
but summaries and tendencies are written in the user's language.

Approved tags (always English): aggro, passive, nit, maniac, fish, reg, 
shark, calling-station, 3bet-happy, slow-player, bluffer, value-heavy, 
overbet, scared-money, tilt-prone, solid, tricky

Return ONLY valid JSON:
{
  "suggestedTags": string[],
  "structuredSummary": string,
  "tendencies": { "preflop": string, "postflop": string },
  "confidence": number
}
`;
}
```

### Storing multi-language AI output
- Store the AI response as-is in whatever language it was generated
- Don't translate on save — would be expensive and lossy
- If user switches locale, show the original note with a "translate" button (future feature)

## 7. Database Considerations

### No schema changes needed
Notes and player descriptions are already `text` — they store whatever language the user wrote. The app doesn't translate user-generated content.

### User preference
Add to `users` table:

```typescript
export const users = sqliteTable("users", {
  // ... existing fields
  preferredLocale: text("preferred_locale", { enum: ["en", "es"] })
    .notNull()
    .default("en"),
});
```

Logged-in users always land on their preferred locale.

## 9. Using Translations

### Server Components (default)

```tsx
// app/[locale]/tools/pot-odds/page.tsx
import { getTranslations } from 'next-intl/server';

export default async function PotOddsPage() {
  const t = await getTranslations('tools.potOdds');

  return (
    <div>
      <h1>{t('title')}</h1>
      <label>{t('potSize')}</label>
    </div>
  );
}

// Metadata also localized
export async function generateMetadata({ params }) {
  const t = await getTranslations('tools.potOdds');
  return {
    title: t('title'),
    description: t('metaDescription'),
  };
}
```

### Client Components

```tsx
'use client';
import { useTranslations } from 'next-intl';

export function AIButton() {
  const t = useTranslations('ai');
  return <button>{t('button')}</button>;
}
```

### Formatting numbers, dates, plurals

```tsx
const t = useTranslations('notes');
t('playerCount', { count: players.length });
// "3 players" / "3 jugadores"
```

## 10. Language Switcher Component

```tsx
// components/LanguageSwitcher.tsx
'use client';
import { useLocale } from 'next-intl';
import { usePathname, useRouter } from '@/i18n/navigation';

export function LanguageSwitcher() {
  const locale = useLocale();
  const pathname = usePathname();
  const router = useRouter();

  function switchTo(next: 'en' | 'es') {
    router.replace(pathname, { locale: next });
    // Also save to cookie/user preference
  }

  return (
    <div role="group" aria-label="Language">
      <button 
        onClick={() => switchTo('en')}
        aria-current={locale === 'en'}
      >EN</button>
      <button 
        onClick={() => switchTo('es')}
        aria-current={locale === 'es'}
      >ES</button>
    </div>
  );
}
```

Position: in settings page + footer on marketing pages.

## 11. SEO Implications

### Hreflang tags (auto-generated)

```tsx
// app/[locale]/layout.tsx
export async function generateMetadata({ params }) {
  return {
    alternates: {
      canonical: `/${params.locale}`,
      languages: {
        'en': '/en',
        'es': '/es',
        'x-default': '/en',
      },
    },
  };
}
```

### Sitemap

```typescript
// app/sitemap.ts
export default function sitemap() {
  const locales = ['en', 'es'];
  const pages = ['', '/tools/pot-odds', '/tools/icm', '/tools/push-fold', '/pricing'];
  
  return locales.flatMap(locale =>
    pages.map(page => ({
      url: `https://pokernotes.app/${locale}${page}`,
      alternates: {
        languages: Object.fromEntries(
          locales.map(l => [l, `https://pokernotes.app/${l}${page}`])
        ),
      },
    }))
  );
}
```

### Keyword strategy per locale

**English targets:**
- "pot odds calculator"
- "icm calculator poker"
- "push fold chart"
- "live poker notes app"

**Spanish targets:**
- "calculadora pot odds"
- "calculadora icm poker"
- "notas de jugadores poker"
- "aplicación poker en vivo"

## 16. Risk: Cloudflare Pages + Edge i18n

### Known limitation
`next-intl` works correctly with OpenNext + Cloudflare Pages, but **verify in staging** that:
- Middleware locale detection works at edge
- Server-rendered translations don't cache incorrectly across locales
- `Accept-Language` header is passed through correctly

### Mitigation
- Sprint 0 end-of-sprint smoke test: verify `/en` and `/es` both render correctly in production Cloudflare deployment
- If issues found, fall back to `localePrefix: 'always'` (no detection logic at edge)
